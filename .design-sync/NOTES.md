# design-sync notes ‚Äî Minecraft Stats UI

Repo-specific gotchas for syncing `frontend/src/components/ui/` (shadcn/Radix +
Tailwind) to claude.ai/design. Read before any re-sync.

## Shape & invocation
- **Package shape, synth-from-src** ‚Äî this is a Next.js *app*, not a published
  component library. There is no `dist/`. We bundle straight from `src/`.
- We DON'T use the converter's auto-synth (it sets `PKG_DIR = node_modules/frontend`,
  which doesn't exist ‚Äî apps don't self-install). Instead a hand-written barrel
  entry `frontend/.ds-entry.mjs` (committed) re-exports every `ui/` file; `cfg.entry`
  points at it. The package.json walk-up from that entry sets `PKG_DIR = frontend`
  correctly, and `--node-modules ./frontend/node_modules` keeps esbuild/react
  resolution working.
- **Do NOT create a `frontend/node_modules/frontend` junction.** It self-recurses
  and ts-morph's directory walk loops forever (`.next/export` deep-path crash).
- Because discovery comes from `.d.ts` (none here ‚Üí empty), the component list is
  driven entirely by `cfg.componentSrcMap` ‚Äî the ~21 top-level primitives. Compound
  sub-parts (DialogContent, SelectTrigger, CommandItem, ‚Ä¶) are NOT carded but ARE on
  `window.MinecraftStatsUI` via the barrel, so compose them inside their parent's
  authored preview.

## Build commands (from repo root)
```
# 1. compile Tailwind to a static stylesheet (cfg.buildCmd) ‚Äî run from frontend/
cd frontend && npx tailwindcss -i src/app/globals.css -o ds-tailwind.css --minify
# 2. converter (from repo root)
node .ds-sync/package-build.mjs --config .design-sync/config.json \
  --node-modules ./frontend/node_modules --out ./ds-bundle
```

## Styling
- Tailwind utilities are generated at build time; `globals.css` alone ships only
  `@tailwind` directives + CSS-variable theme. We compile a static stylesheet
  (`frontend/ds-tailwind.css`, gitignored, ~85 KB) and point `cfg.cssEntry` at it.
  **Re-sync MUST re-run the Tailwind compile** (cfg.buildCmd) or previews render
  unstyled.
- Theme: CSS variables on `:root` (light) and `.dark`. Previews render light mode.

## Collision
- `multi-select.tsx` and `fancy-multi-select.tsx` BOTH export `FancyMultiSelect`.
  The barrel excludes `multi-select.tsx`; we ship the richer `fancy-multi-select.tsx`.

## Known render warns
- `[RENDER_BLANK]` on a carded component just means its rich preview hasn't been
  authored yet (floor card). Not a failure.

## Converter harness patches (CRITICAL ‚Äî re-apply after re-copying `.ds-sync/`)
cmdk ships a literal diacritic-stripping regex `/[ÃÄ-ÕØ]/g` as raw UTF-8
bytes. The bundle is valid UTF-8 and works everywhere it's loaded as UTF-8 (the
real app + the preview cards, which declare `<meta charset="utf-8">`). BUT two
converter harness spots load the bundle WITHOUT declaring UTF-8, so Chromium
decodes those bytes as Latin-1 ‚Üí `SyntaxError: Invalid regular expression ‚Ä¶ Range
out of order` ‚Üí `window.MinecraftStatsUI` never defines ‚Üí false `[BUNDLE_EXPORT]
20/20 not a component`. Fix (both are gitignored staged copies ‚Äî REDO each re-sync
after the `cp -r` that stages scripts):
1. `.ds-sync/storybook/http-serve.mjs` line ~12 MIME: add `; charset=utf-8` to the
   `.html/.js/.mjs/.css/.json` types. (Primary fix ‚Äî serves JS as UTF-8.)
2. `.ds-sync/package-validate.mjs` smoke check (~line 455): add `<meta charset="utf-8">`
   right after `<!doctype html>` in the `setContent(...)` string (mirrors the
   per-preview harness at ~line 757).
Verify: `node .ds-sync/package-validate.mjs ./ds-bundle` prints
`window.MinecraftStatsUI: 90 exports` and exits 0.

## Re-sync risks
- `frontend/ds-tailwind.css` is gitignored and regenerated by cfg.buildCmd ‚Äî if the
  compile step is skipped, every preview loses its utility classes silently.
- `frontend/.ds-entry.mjs` is committed; if a new `ui/` component is added, add it to
  the barrel AND to `cfg.componentSrcMap`, or it won't be carded.
- Playwright/Chromium installed into `.ds-sync/node_modules` (gitignored) ‚Äî re-install
  on a fresh clone for the render check.
- `Avatar.tsx` preview uses a dicebear identicon URL (https://api.dicebear.com/...) for the
  image cell ‚Äî network-dependent at capture time. It also has a fallback-only cell that is
  network-independent, so a dicebear outage degrades one cell, not the component.
- Overlay components (Dialog, Popover, Tooltip, DropdownMenu, Toast) use `cfg.overrides.<Name>`
  `{cardMode:"single", viewport, primaryStory}` and render their OPEN state (Radix `open` prop)
  so the portaled/fixed content is captured. Select/Command/FancyMultiSelect/Form render inline.
- Toaster is a runtime mount point (renders `useToast()` toasts, empty statically) ‚Üí it ships
  a floor card by design. Toast itself has the authored open-state preview.
- Toast preview gotcha: the real `ToastViewport` is `position:fixed` (out of flow), which
  collapses the preview card to zero height ‚Üí `[RENDER_BLANK]`. The preview overrides
  `ToastViewport` with inline `position:static` (in-flow) and sets `animation:none` on each
  Toast (the slide-in enter animation otherwise starts off-screen). Keep these if reworking.
